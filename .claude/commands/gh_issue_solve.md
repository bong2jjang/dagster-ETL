# GitHub 이슈 분석 및 해결

GitHub 이슈를 분석하고, 실행 가능한지 검증한 후, 구현 계획을 제안합니다.

## 사용법

```bash
/gh_issue_solve <이슈_번호>
```

## 실행 단계

1. **이슈 세부 정보 조회**: `gh api`로 이슈 내용을 가져옵니다.
   - 제목, 설명, 라벨, 담당자
   - 댓글 및 토론 컨텍스트
   - 관련 PR 또는 이슈

2. **이슈 유형 판단**:
   - **유효한 버그**: 재현 가능한 문제, 명확한 증상
   - **유효한 기능 요청**: 명확히 정의된 개선/신규 기능
   - **불명확/무효**: 추가 정보 필요, 중복, 범위 밖
   - **이미 해결**: 기존 PR 또는 해결 완료

3. **코드베이스 분석**: 유효한 이슈의 경우
   - grep/glob으로 관련 코드 탐색
   - 기존 구현 검토
   - 관련 테스트 확인
   - 영향받는 패키지/모듈 식별

4. **구현 제안**:
   - 근본 원인 분석 (버그의 경우)
   - 구현 접근법
   - 변경 필요 파일 목록
   - 테스트 전략
   - 잠재적 엣지 케이스

5. **할 일 목록 생성**: 구현을 실행 가능한 단계로 분해

## 출력 형식

### 유효한 이슈

```markdown
## 이슈 분석: #번호 - [이슈 제목]

**유형**: 버그 | 기능 요청
**상태**: 유효하고 실행 가능
**영향 영역**: [예: Asset 시스템, 팩토리, dbt 통합]

### 요약
[이슈의 간결한 설명]

### 근본 원인 (버그의 경우)
[기술적 설명]

### 구현 제안

**변경 파일**:
- `path/to/file.py:123` - [변경 내용]

**접근법**: [단계별 기술적 접근]
**테스트 전략**: [검증 방법]
**주의사항**: [엣지 케이스 또는 위험]
```

### 불명확한 이슈

```markdown
## 이슈 분석: #번호 - [이슈 제목]

**상태**: 추가 정보 필요

**부족한 정보**: [필요한 세부 사항]
**제안 응답**: [추가 정보 요청 템플릿]
```

## 개발 워크플로우 연계

분석 후 다음을 준비합니다:
1. 새 브랜치 생성: `git checkout -b fix-issue-번호`
2. 할 일 목록에 따라 구현
3. Python 변경 후 `ruff check --fix . && ruff format .` 실행 (필수)
4. 테스트 실행: `pytest tests/`
5. 이슈를 링크한 PR 생성
